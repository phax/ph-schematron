# Delta: ISO/IEC 19757-3:2020 → ISO/IEC 19757-3:2025

**Source A (v3)**: ISO/IEC 19757-3:2020 (Third edition, 2020-06)
**Source B (v4)**: ISO/IEC 19757-3:2025 (Fourth edition, 2025-09)
**Date**: 2026-06-06

This document is a section-by-section, element-by-element delta derived from the markdown conversions of both standards in `/Users/philip/dev/git/ph-schematron/docs/`. It is intended as a migration reference for Schematron implementers and tool authors.

---

## 1. High-level summary

ISO/IEC 19757-3:2025 is a *technical revision* of the 2020 edition. The committee explicitly states the change is technical, not editorial, and that "this edition is backwards compatible … to the extent that validation results produced by existing schemas are unaffected by the changes this edition makes".

The dominant themes of the revision are:

1. **Three new schema-language constructs**: top-level `group`, top-level `library` and top-level `rules` containers; `param` at schema and abstract-pattern level.
2. **Dynamic attribute values**: `flag`, `role` and the new `severity` are dynamically evaluated when their value is a variable reference (so a single schema can vary severity by selected phase).
3. **Datatyping of variables**: the new `as` attribute on `let`.
4. **Dynamic phase selection**: new `when` attribute on `phase`, new `#ANY` reserved phase string.
5. **Document slicing**: new `from` attribute on `phase`; new `visit-each` attribute on `rule`.
6. **Two new query language bindings**: XQuery 3.0 (Annex M) and XQuery 3.1 (Annex N); a previously missing XPath 3.1 binding is also added (Annex L).
7. **SVRL becomes formally mapped to schema structures** in a new Annex D.2 table, and SVRL itself grows new constructs: `svrl:error`, `svrl:active-group`, `@schemaFrontEdition`, `@visit-each`, `@severity`, list-typed `flag`, and a richer `schematron-output` content model.
8. **Self-identifying schemas**: new `schematronEdition` attribute on `schema` whose value is `2025`.
9. **Explicit fixup rules** for `xml:lang` and `xml:base` carried over from XInclude.
10. **Annex L (EXSLT) and Annex M (STX) are dropped**; Annex K becomes XPath 3.0, and Annexes L, M, N now host XPath 3.1, XQuery 3.0 and XQuery 3.1 respectively. Annex N (properties examples) is renumbered Annex O.

The number of normative annexes grows from 5 to 8; the number of informative annexes shrinks from 7 to 4.

### 2.8 Clause 6 — Semantics

#### 6.1 Validation function

**Changed**: The list of inputs gains `#ANY` as a third reserved phase name. Otherwise unchanged.

#### 6.2 Minimal syntax

In v3 this clause was tagged *informative*. **In v4 the "(informative)" qualifier is dropped** (so it is now part of the normative semantics). The list of XPaths "not in the minimal syntax" gains three new entries:

```
- //sch:group[@abstract="true"]
- //sch:group[@is-a]
- //sch:rules
```

#### 6.3 (new) Processing model

Entire **new clause 6.3** specifies a high-level processing model and ties schema structures to SVRL output. It includes worked examples of which rules fire / do not fire in the presence of multiple rule contexts in a pattern. Also explicitly contrasts `pattern` (if-then-else) vs `group` (try every rule) semantics.

The v3 clause 6.3 "Abstract pattern processing" is renumbered **6.4** in v4.

#### 6.4 Abstract pattern processing (v3: 6.3)

**Changed**: The introductory example now shows the abstract pattern with explicit `<sch:param name="..."/>` declarations (without values) — illustrating the new schema-level / abstract-pattern `param` mechanism.

#### 6.5 Query language binding (v3: 6.4)

**Changed**: The list of reserved query-language names is extended:

```
v3: stx, xslt, exslt, xslt2, xslt3, xpath, xpath2, xpath3, xpath31, xquery, xquery3, xquery31
v4: stx, xslt, exslt, xslt2, xslt3, xslt4, xpath, xpath2, xpath3, xpath31, xpath4,
    xquery, xquery3, xquery31, xquery4
```

I.e. `xslt4`, `xpath4`, `xquery4` are newly reserved. The mapping of bindings to annexes is updated for the new annex layout (H/I/J/K/L/M/N). The sentence "See Annexes L and M for additional information regarding other query language bindings" (referring to the informative EXSLT/STX annexes) is **removed** because those annexes are gone.

#### 6.6 Order and side-effects (v3: 6.5)

**Changed**: A new sentence clarifies that the if-then-else rule semantics applies only inside a `pattern`; inside a `group`, every rule's context is matched independently.

#### 6.7 (new) Localization and language fixup

Entire **new clause**. Mandates `xml:lang` fixup per section 4.5.6 of XInclude 1.0 when resolving `include`, `extends` and instantiating abstract patterns/rules/diagnostics/properties.

#### 6.8 (new) Base URI fixup

Entire **new clause**. Mandates `xml:base` fixup per section 4.5.5 of XInclude 1.0 when resolving `include` and `extends`.

### 2.9 Clause 7 — Conformance

**Changed**:

- Reference to "the ISO Schematron schema" in Annex B is simplified to "the constraints of Annex B".
- The encouragement in v3's Note 1 ("implementations are encouraged to report this information") is promoted to a normative sentence: "Where the query language allows incorrectness to be established, implementations *should* report this information as part of the validation."
- Otherwise the two-tier (simple / full) conformance model is unchanged.

### 2.10 Annexes — overall structure

The annex layout has been substantially reshaped. See **§7 Annex delta** for the full mapping.

## 5. RNC schema delta (Annex A)

Both editions ship an ISO/IEC 19757-2 (RELAX NG Compact Syntax) schema in Annex A. The v4 schema has the © ISO/IEC **2024** header (v3: 2017). Below is the substantive delta.

### 5.1 New top-level production

```rnc
# v4 only
chart = schema | include
```

v3 declared `start = schema`; v4 adds `chart` as a new top-level alternative (`schema` *or* `include`). (The header still declares `start = schema` for the main schema, but the `chart` production is now also defined — note this also signals the SDK can be entered through an `include`-rooted document.)

### 5.2 Changes to `schema`

```rnc
# v4
schema =
    element schema {
        attribute id { xsd:ID }?,
        rich,
        attribute schemaVersion { non-empty-string }?,
        attribute schematronEdition { non-empty-string }?,        # NEW
        attribute defaultPhase { xsd:IDREF }?,
        attribute queryBinding { non-empty-string }?,
        (foreign
        & (inclusion | extends)*                                  # extends now allowed
        & (title?, ns*, p*, param*, let*, phase*,                 # param* is NEW
           abstract-rules*,                                       # NEW (top-level rules elements)
           (rule-set|pattern)+,                                   # group OR pattern (NEW)
           p*, diagnostics?, properties?))
    }
```

vs. v3:

```rnc
# v3
schema =
    element schema {
         attribute id { xsd:ID }?,
         rich,
         attribute schemaVersion { non-empty-string }?,
         attribute defaultPhase { xsd:IDREF }?,
         attribute queryBinding { non-empty-string }?,
         (foreign
          & inclusion*
          & (title?, ns*, p*, let*, phase*, pattern+, p*, diagnostics?, properties?))
    }
```

Key differences:

- New `schematronEdition` attribute.
- `inclusion*` content slot becomes `(inclusion | extends)*` — `extends` allowed at top level.
- New `param*` slot.
- New `abstract-rules*` slot (i.e. zero or more `<sch:rules>` containers).
- The required `pattern+` is replaced by `(rule-set|pattern)+` — at least one pattern *or* group is required.

### 5.3 Changes to `assert` / `report` (shared)

```rnc
# v4
assert =
    element assert {
        attribute test { exprValue },
        attribute flag { flagValue }?,
        attribute id { xsd:ID }?,
        attribute diagnostics { xsd:IDREFS }?,
        attribute properties { xsd:IDREFS }?,
        rich,
        linkable,
        severity,                                                  # NEW pattern
        message                                                    # NEW content model
    }
severity = attribute severity { text }?                            # NEW
message = (foreign & (text | dynamic | emph | dir | span)*)        # NEW
dynamic = value-of | name                                          # NEW
```

vs. v3 inlined `(foreign & (text | name | value-of | emph | dir | span)*)`. Functionally the message content is the same set of children, but it is now factored into a reusable `message` production and a `dynamic` group, and the new `severity` attribute is allowed on `assert`, `report` and (per the RNC) `diagnostic`/`property` content via the `message` group.

### 5.4 Changes to `let`

```rnc
# v4
let =
    element let {
        attribute name { nameValue },
        attribute as { text }?,        # NEW
        (attribute value { string } | foreign-element+)
    }
```

### 5.5 New `library` element

```rnc
# v4 only
library =
    element library {
        attribute id { xsd:ID }?,
        rich,
        (foreign
        & (inclusion | extends)*
        & (title?, p*, param*, let*, abstract-rules*, (rule-set|pattern)*,
           p*, diagnostics?, properties?))
    }
```

### 5.6 Changes to `pattern` / new `group` (`rule-set`)

The content model that v3 inlined into `pattern` is factored out into a reusable `rule-set-or-pattern` production in v4. Both `pattern` and the new `group` (RNC name `rule-set`) reuse it:

```rnc
# v4
pattern = element pattern { rule-set-or-pattern }
rule-set = element group { rule-set-or-pattern }

rule-set-or-pattern =
    attribute documents { pathValue }?,
    attribute role { roleValue }?,                                 # role on pattern/group: NEW
    rich,
    (foreign
    & inclusion*
    & ((attribute abstract { "true" },
        attribute id { xsd:ID },
        title?,
        (p*, param*, let*, rule*))                                 # let* and param* NEW in abstract pattern
    | (attribute abstract { "false" }?,
        attribute id { xsd:ID }?,
        title?,
        (p*, let*, rule*))
    | (attribute abstract { xsd:NMTOKEN },                         # was { "false" }?
        attribute is-a { xsd:IDREF },
        attribute id { xsd:ID }?,
        title?,
        (p*, param*))))
```

Notable changes:

- `role` is now an attribute of `pattern`/`group` directly (no longer indirectly via `linkable`).
- Abstract pattern (`abstract="true"` branch) gains `param*` and `let*` slots.
- The `is-a` branch's `abstract` attribute changes from optional `"false"` to `xsd:NMTOKEN` (i.e. any token value permitted) — this allows the `abstract` attribute to be present with a non-`"false"` value on `is-a` patterns.

### 5.7 Changes to `rule`

```rnc
# v4 — concrete rule
rule =
    element rule {
        attribute flag { flagValue }?,
        rich,
        linkable,
        (foreign
        & inclusion*
        & ((attribute context { pathValue },
            attribute visit-each { non-empty-string }?,            # NEW
            attribute id { xsd:ID }?,
            attribute abstract { "false" }?,
            title?,
            let*,
            (assert | report | extends | p)+)))
    }

# v4 — abstract rule, now a separate production
abstract-rule =
    element rule {
        attribute flag { flagValue }?,
        rich,
        linkable,
        (foreign
        & inclusion*
        & (attribute abstract { "true" },
           attribute id { xsd:ID },
           title?,
           let*,
           (assert | report | extends | p)+))
    }

# NEW container
abstract-rules =
    element rules {
        attribute id { xsd:ID }?,
        rich,
        title?,
        p*,
        abstract-rule+
    }
```

v3 inlined both concrete and abstract rule shapes into a single `rule` production. v4 splits them, introduces `visit-each` on concrete rules, and adds `abstract-rules` (i.e. `<sch:rules>`) as a container for `abstract-rule+`.

### 5.9 Changes to `param`

```rnc
# v4
param =
    element param {
        attribute name { nameValue },
        attribute value { non-empty-string }?     # was required; now optional
    }
```

The `value` attribute is now optional. (The narrative clause 5.4.11 specifies the rules for when it is required, vs. when it may be defaulted from the abstract pattern.)

### 5.10 Changes to `dir`, `emph`, `span`

These three previously had narrow content models (mostly `text`). In v4 they all allow `dynamic` content (i.e. `value-of` and `name`), and `dir`/`span` additionally allow `foreign`:

```rnc
# v4
dir   = element dir   { attribute value { "ltr" | "rtl" }?, (foreign & (text | dynamic)*) }
emph  = element emph  { (foreign & (text | dynamic)*) }
span  = element span  { attribute class { classValue }, (foreign & (text | dynamic)*) }
```

vs. v3, where `dir = element dir { attribute value { "ltr" | "rtl" }?, (foreign & text) }`, `emph = element emph { text }`, and `span` allowed only `text` plus `foreign`.

### 5.11 Changes to `diagnostic` / `property`

Both now use the new shared `message` group:

```rnc
# v4
diagnostic = element diagnostic { attribute id { xsd:ID }, attribute role { roleValue }?, rich, message }
property   = element property   { attribute id { xsd:ID }, attribute role { roleValue }?, attribute scheme { text }?, message }
```

(In v3 they expanded the same set of children inline as `(foreign & (text | name | value-of | emph | dir | span)*)` — the v4 form is logically equivalent for these two elements, but factored.)

### 5.12 Datatype changes

| Datatype name | v3 | v4 | Note |
|---|---|---|---|
| `flagValue` | `string` | `list { token+ }` | A single `@flag` may now carry multiple flag names (whitespace-separated tokens) |
| `pathValue` | `string` | `string` | unchanged |
| `uriValue` | `xsd:anyURI` | `xsd:anyURI` | unchanged |
| Others | unchanged | unchanged | — |

### 5.13 Summary of removed RNC productions

None. Every v3 production has a corresponding v4 production (sometimes renamed: `rule` is split into `rule`+`abstract-rule`; `pattern`'s inlined content model is renamed `rule-set-or-pattern`).

### 6.2 SVRL element / attribute changes

**Added to SVRL** in v4:

- `svrl:error` — top-level errors raised by the implementation (`@document`, `@location`, `@code`, rich-text content).
- `svrl:active-group` — counterpart to `svrl:active-pattern` for the new `sch:group`.
- `svrl:schematron-output/@schemaFrontEdition` — reports which Schematron edition produced the report.
- `svrl:fired-rule/@visit-each` — reports the rule's `visit-each` expression.
- `svrl:fired-rule/@flag` datatype changes from a scalar `string` to `list { token+ }`.
- `svrl:diagnostic-reference/@role` — newly permitted.
- `attlist.assert-and-report/@severity` — newly permitted.
- `attlist.assert-and-report/@flag` datatype changes to `list { token+ }`.

**Structural change to `svrl:schematron-output`**:

v3 content model:
```rnc
human-text*,
ns-prefix-in-attribute-values*,
(active-pattern,
 (fired-rule, (failed-assert | successful-report)*)*)+
```

v4 content model:
```rnc
(
  (error*, human-text*)
  |
  (error*,
   human-text*,
   ns-prefix-in-attribute-values*,
   (active-pattern+, rule-set,
    (fired-rule, (failed-assert | successful-report)*)*)*
  )
)
```

The new shape allows pure-error reports (no patterns), interleaves `svrl:error` everywhere, and admits `svrl:active-group` (`rule-set`) entries next to `svrl:active-pattern`.

**`svrl:active-pattern` / `svrl:active-group`**:

v3 inlined attributes (`id`, `documents`, `name`, `role`). v4 factors them out into a reusable `active` production that also allows `error*` content, so subordinate-document retrieval errors can be reported per active pattern.

**`svrl:failed-assert` / `svrl:successful-report`**:

v3 inlined `attlist.assert-and-report, diagnostic-reference*, property-reference*, human-text`. v4 wraps these in an `assertion-message` production with the same content but adds `@severity`. The list-typed `@flag` change also applies here.

**`diagnostic-reference`**: v3 had `@diagnostic` and `human-text`. v4 adds `@role`.

**Removed from SVRL in v4**: nothing — every v3 SVRL element is retained.

### 6.3 Annex D.4 Schematron schema (informative)

The example schema validating SVRL has been updated to match the new content model:

- The "Catch-all rule" (`<sch:rule context="*">` that reported "An unknown … element has been used") is **removed**.
- The combined `svrl:text` rule is split.
- The example pattern `requiredAttribute` is restructured (changed `$attribute /name()` form, etc.).

### 6.4 D.2 Mapping table

The new D.2 table (≈45 rows + 6 footnotes) is the most prominent new piece of normative-style content in Annex D (Annex D itself remains informative). It includes mappings for `@defaultPhase`, `@id`, `@queryBinding`, `@schemaVersion`, `@schemaFrontVersion`, `sch:diagnostics/sch:diagnostic` and its `@id`, `sch:group/@documents|@id|@role`, every `sch:rule/@*`, `sch:assert/@*` and `sch:report/@*` including `@severity`, `sch:properties/sch:property` and its attributes, generic `dir`/`emph`/`name`/`span`/`value-of`, and explicit mappings for errors encountered while evaluating `pattern/@documents` (→ `svrl:active-pattern/@documents/svrl:error`) and for dynamic errors during evaluation (→ `svrl:schematron-output/svrl:error`).

## 8. Migration notes (practical impact for implementers)

The following bullets summarise the concrete impact on a Schematron implementation upgrading from a 2020-conformant validator to a 2025-conformant validator. Items are grouped by area.

### Schema parsing (Annex A grammar)

- **Accept new root**: `library` is now an allowed root element, in addition to `schema`.
- **Accept new top-level children of `schema`**: `param*`, `rules*`, `group*`, top-level `extends*`.
- **Accept new optional attribute on `schema`**: `schematronEdition` (only `2025` is currently valid).
- **Accept `group` everywhere `pattern` is accepted**: same content model, but different rule semantics at validation time (see Processing model below).
- **Accept `rules`** containers carrying `<rule abstract='true'>` children.
- **Accept `as` attribute on `let`**.
- **Accept `from` and `when` attributes on `phase`**, and the new reserved phase name `#ANY`.
- **Accept `visit-each` attribute on `rule`**.
- **Accept `severity` attribute on `assert`, `report`, `rule`** (reserved values: `fatal`, `error`, `warning`, `info`).
- **Accept multi-token `@flag`** values (the RNC datatype changes from string to list of tokens).
- **Accept `param` with no `@value`** (the value can be supplied from the abstract pattern's `param` defaults; for schema params it can be overridden at user option).
- **Accept richer content** in `dir`, `emph`, `span` (now allow embedded `value-of` and `name`, and `foreign` markup).

### Schema processing semantics

- **Implement `group` semantics**: every contained `rule` should attempt to match independently of previously-fired rules in the same `group` (no if-then-else).
- **Implement schema- and abstract-pattern-level parameters**: schema `param` declares an external variable (overridable at user option, mechanism implementation-defined). Abstract-pattern `param` may carry a default value that is overridden by a concrete pattern's `param`.
- **Implement `extends` at the schema/library level** (used to include external `library` content) and `extends/@href` for non-abstract rules.
- **Implement language fixup (`xml:lang`)** per XInclude 1.0 §4.5.6 when resolving `include`, `extends`, abstract patterns, abstract rules, diagnostics and properties.
- **Implement base URI fixup (`xml:base`)** per XInclude 1.0 §4.5.5 when resolving `include` and `extends`.
- **Implement dynamic phase selection**: if the active phase is `#ANY`, evaluate each phase's `@when` (Boolean, in the root context) in document order; the first whose expression is `true()` becomes active. If none matches, behave as `#ALL`.
- **Implement `phase/@from`**: when present, evaluate the expression against the root and use the resulting items as the context for rule context expressions within that phase. Does not apply to subordinate documents (`pattern/@documents`).
- **Implement `rule/@visit-each`**: when present, evaluate against each rule-context item; if non-empty, use the result as the context for assertions in that rule; rule is still "fired" even if the expression is empty.
- **Implement dynamic attribute values for `flag`, `role`, `severity`**: when the value is a variable reference, evaluate it dynamically (variable scope is what is in scope at the assert/report site).
- **Error reporting**: errors during processing of `include`, `extends`, and unretrievable subordinate documents (`pattern/@documents`) shall be reported; termination behaviour is implementation-defined. Output via `svrl:error` is one suggested mechanism.

### SVRL output

- Emit **`svrl:schematron-output/@schemaFrontEdition`** (reports edition the schema declared via `schematronEdition`).
- Emit **`svrl:active-group`** for active `group`s.
- Emit **`svrl:error`** for dynamic errors (top-level) and **`svrl:active-pattern/.../svrl:error`** for per-pattern errors. Provide `@document`, `@location`, optional `@code`.
- Emit **`svrl:fired-rule/@visit-each`** when the source `rule` used `visit-each`.
- Emit **`svrl:fired-rule/@flag`** and **`svrl:failed-assert/@flag`** as space-separated lists of tokens.
- Emit **`svrl:failed-assert/@severity`** and **`svrl:successful-report/@severity`**.
- Emit **`svrl:diagnostic-reference/@role`** when present on the diagnostic.
- Be prepared to emit a `schematron-output` with no `active-pattern` children — only `error*` and `human-text*` — when validation could not start (per the new content model).

### Query language bindings

- Add **`xquery3`** and **`xquery31`** bindings (Annexes M, N): rule context evaluated relative to the root; context may also include text nodes; `as` is interpreted per XQuery3 Production 167 / XQuery31 Production 183.
- Add **`xpath31`** binding (Annex L).
- Update **`xslt2`** (Annex H): allow `xsl:import`, `xsl:include`, `xsl:import-schema` before patterns; interpret `let/@as` per XSLT2 §9.1.
- Update **`xslt3`** (Annex J): allow `xsl:import`, `xsl:include`, `xsl:use-package`, `xsl:import-schema`, `xsl:accumulator` before patterns; interpret `let/@as` per XSLT3 §9.1; support text value templates and attribute value templates for `flag`/`role`/`severity`.
- Update **default `xslt`** binding (Annex C): allow `xsl:import` and `xsl:include` before patterns; ignore `let/@as`; allow Schematron value templates for `flag`/`role`/`severity`.
- **Drop or move EXSLT and STX bindings** if they were ever implemented from the 2020 informative annexes — they are no longer described in the 2025 standard.
- Recognise the additional reserved binding names `xslt4`, `xpath4`, `xquery4` (no semantics defined yet, but reserved).

### Conformance reporting

- Both "simple" and "full" conformance now formally require implementations to report query-language-detected incorrectness *where the query language allows* — promoted from a v3 NOTE to a SHOULD in v4.

## Notes on ambiguities encountered

- The v4 markdown shows some OCR-level imperfections (e.g. `1ibrary`, `1et`, `5.4.3.  assert` with a dot, `Library` mis-capitalised in the Foreword change list). The underlying intent matches the RNC schema and surrounding prose, so this delta uses the canonical element/attribute names from the Annex A RNC.
- v4 Annex C says XSLT1 `import` and `include` may be used before patterns — v3 mentioned only the XSLT1 `key` element here. This is a real, deliberate extension.
- v4 Annex J text says "A Schematron `let` expression shall be treated as an **XQuery3** variable" — this looks like an editorial error (should presumably be "XSLT3 variable"). Implementers should treat it as an XSLT3 variable with optional `as` declaration as per XSLT3 §9.1.
- The v4 RNC introduces a top-level production `chart = schema | include` but the schema still declares `start = schema`. This is a minor inconsistency; implementers can safely continue to use `schema` as the only validating start.
- The Annex B Schematron-of-Schematron in v4 contains what appears to be a parsing artifact mid-pattern (a stray `<sch:rule>` closing tag before the `<sch:rule context="sch:pattern[@is-a]/sch:param">` rule). The intent — to validate that `is-a` references resolve, that `param` names line up between abstract and concrete patterns, that pattern parameter values are supplied either directly or by default, and that schema parameters have a `value` — is unambiguous, but implementers may want to consult the official PDF for the precise XML.
- v4 specifies `schematronEdition` must equal `2025` when present. That is an absolute constraint; future editions presumably will reserve a new value. Implementations should warn rather than error on other values to remain forward-tolerant.
