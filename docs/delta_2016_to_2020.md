# Delta: ISO/IEC 19757-3:2016 vs ISO/IEC 19757-3:2020

**Comparison of:**

- **Source A (v2)**: ISO/IEC 19757-3:2016(E) — *Second edition, 2016-01-15* — file `c055982_ISO_IEC_19757-3_2016.md`
- **Target B (v3)**: ISO/IEC 19757-3:2020(E) — *Third edition, 2020-06* — file `ISO_IEC_19757-3_2020_en.md`

**Delta produced:** 2026-06-06

---

## 1 High-level summary

The 2020 edition is a **technical revision** of the 2016 edition, not a re-architecture. The Foreword of the 2020 edition explicitly states:

> "This third edition cancels and replaces the second edition (ISO/IEC 19757-3:2016), which has been technically revised."

and limits the headline changes to:

> "— query language bindings have been added for XSLT 3.0 (Annex J) and XPath 3.0 (Annex K);
> — annexes pertaining to XPath and XSLT query language bindings (Annexes H to K) are now all normative, while those for EXSLT (Annex L) and STX (Annex M) remain informative."

The Introduction confirms backwards compatibility:

> "This edition is backwards compatible with ISO/IEC 19757-3:2016, supersedes it and provides extra query language bindings, in particular for XSLT3."

The dominant themes of change are therefore:

1. **Query language coverage** — first-class normative bindings for XSLT 3.0 / XPath 3.0; the previously informative XPath 2.0 binding is promoted to normative.
2. **Editorial restructuring of Clause 5.4** — a new sub-clause `5.4.1 General`, plus rewording of every element description in declarative ("The X element …") form. The numbering of the sub-clauses 5.4.2 onwards therefore shifts by one against the 2016 edition.
3. **Tightening of terms and definitions** — several 2016 terms are dropped (`compound document`, `correct schema`, `good schema`, `schema`); the term `correct schema` is replaced with `Schematron schema`, and the `name` definition is reworked.
4. **RELAX NG schema (Annex A) tweaks** — `properties` is now optional on `schema` (`properties?` instead of mandatory `properties`); the `diagnostic` element acquires a `role` attribute.
5. **SVRL schema (Annex D) tweaks** — `fired-rule` acquires a new `document` attribute; `xsd:ID` / `xsd:NMTOKEN` typing of `id` and `role` attributes is loosened to `xsd:NCName` / `string`; the content model of `schematron-output` is widened to allow zero `fired-rule` per active pattern.
6. **6.2 Minimal syntax is now informative**, no longer normative (the heading is explicitly tagged "(informative)").
7. **Conformance and semantics editorial rewording** — Clause 7 is rewritten to refer to a "Schematron schema" rather than to a "valid schema" / "correct schema".

No new core Schematron element or attribute is introduced. No element or attribute is removed. The 2016 edition's NOTE markers calling out features that were "not part of ISO/IEC 19757-3:2006" have been removed from the 2020 element descriptions.

**Unchanged in substance** — the four "Considered as / theoretically / analytically / based on Schematron[2]" paragraphs are essentially identical.

### 2.8 Clause 7 — Conformance

**Changed**

- 2016 §7.1 (simple conformance) referred to "a valid schema [that] conforms to the constraints of Annex A". The 2020 §7.1 replaces these with "A Schematron schema shall conform to the constraints of Annex A" and **adds** a second bullet for Annex B:

  > "— A Schematron schema shall conform to the constraints of Annex B, the ISO Schematron schema."

  This makes Annex B's additional constraints a simple-conformance requirement too (in 2016, only full-conformance §7.2 cited Annex B).
- 2016 §7.1 phrased the simple-conformance ability as "shall be able to report for any XML document that its structure does not conform to that of a valid Schematron schema"; 2020 rewords this to: "shall be able to report for any XML document that it does not conform to the constraints expressed by a given Schematron schema." The 2020 wording shifts the obligation from *schema-structure validation* to *instance-document validation* against a schema.
- 2016 §7.2 (full conformance) full-conformance bullet 4 was: "A correct schema has one definition only in scope for any variable name in any context." The 2020 §7.2 wording is more precise: "A Schematron schema shall have one definition only in scope for any global variable name in the global context and any local variable name in the local context." (Variable scoping is now split into global vs local.)

## 5 RNC schema delta

This section captures every observable difference between the RELAX NG Compact Syntax schemas in **Annex A** of the 2016 and 2020 editions.

### 5.1 Copyright header

- 2016 header reads `Copyright © ISO/IEC 2015`.
- 2020 header reads `Copyright © ISO/IEC 2017`.

### 5.2 `schema` element — content model

**2016 (Annex A, lines 717–727):**

```rnc
schema =
  element schema {
    attribute id { xsd:ID }?,
    rich,
    attribute schemaVersion { non-empty-string }?,
    attribute defaultPhase { xsd:IDREF }?,
    attribute queryBinding { non-empty-string }?,
    (foreign
     & inclusion*
     & (title?, ns*, p*, let*, phase*, pattern+, p*, diagnostics?, properties))
  }
```

**2020 (Annex A, lines 908–918):**

```rnc
schema =
    element schema {
         attribute id { xsd:ID }?,
         rich,
         attribute schemaVersion { non-empty-string }?,
         attribute defaultPhase { xsd:IDREF }?,
         attribute queryBinding { non-empty-string }?,
         (foreign
          & inclusion*
          & (title?, ns*, p*, let*, phase*, pattern+, p*, diagnostics?, properties?))
    }
```

**Diff**: the final child production changes from `properties` to `properties?` — i.e. the `properties` element is now optional inside `schema`. This corrects what was almost certainly a typo in the 2016 RNC (the prose of 2016 §5.4.13 never says `properties` is mandatory).

### 5.3 `diagnostic` element — new `role` attribute

**2016:**

```rnc
diagnostic =
  element diagnostic {
    attribute id { xsd:ID },
    rich,
    (foreign & (text | value-of | emph | dir | span)*)
  }
```

**2020:**

```rnc
diagnostic =
     element diagnostic {
         attribute id { xsd:ID },
         attribute role { roleValue }?,
         rich,
         (foreign & (text | value-of | emph | dir | span)*)
     }
```

**Diff**: 2020 adds an optional `role { roleValue }?` attribute to `diagnostic`. This formalises the 2016 §5.5.1 NOTE 2 hint ("Typical values for the `role` attribute on a `diagnostic` element might be `warning`, `caution` or `note`") which previously had no RNC backing.

### 5.4 All other productions

All remaining RNC productions (`active`, `assert`, `diagnostics`, `dir`, `emph`, `extends`, `let`, `name`, `ns`, `p`, `param`, `pattern`, `phase`, `properties`, `property`, `report`, `rule`, `span`, `title`, `value-of`, `inclusion`, `rich`, `linkable`, `foreign`, `foreign-empty`, `foreign-attributes`, `foreign-element`) are **bit-identical in content** between the two editions (only indentation and whitespace differ).

### 5.5 Data type declarations

Identical:

```rnc
uriValue = xsd:anyURI
pathValue = string
exprValue = string
fpiValue = string
langValue = xsd:language
roleValue = string
flagValue = string
nameValue = string
# In the default query language binding, xsd:NCNAME
classValue = string
non-empty-string = xsd:token { minLength = "1" }
```

(2020 places the `# In the default query language binding, xsd:NCNAME` comment after `nameValue` rather than between `nameValue` and `classValue`, but it is the same comment about the same production.)

### 5.7 SVRL RNC schema delta (Annex D.2)

The SVRL grammar is informative in both editions but is significantly tweaked in 2020.

**2016 (excerpt):**

```rnc
schematron-output =
  element schematron-output {
    attribute title { text }?,
    attribute phase { xsd:NMTOKEN }?,
    attribute schemaVersion { text }?,
    human-text*,
    ns-prefix-in-attribute-values*,
    (active-pattern,
     (fired-rule, (failed-assert | successful-report)*)+)+
  }
```

**2020 (excerpt):**

```rnc
schematron-output =
    element schematron-output {
        attribute title { text }?,
        attribute phase { xsd:NMTOKEN }?,
        attribute schemaVersion { text }?,
        human-text*,
        ns-prefix-in-attribute-values*,
        (active-pattern,
         (fired-rule, (failed-assert | successful-report)*)*)+
    }
```

**Diff**: the inner `(fired-rule, …)+` becomes `(fired-rule, …)*` — i.e. an active pattern may now legitimately contain zero `fired-rule` children (useful when no rule fires for that active pattern).

**`active-pattern`:**

| Attribute | 2016 type | 2020 type |
|---|---|---|
| `id` | `xsd:ID` | `xsd:NCName` |
| `documents` | `text` | `text` |
| `name` | `text` | `text` |
| `role` | `xsd:NMTOKEN` | `string` |

**`fired-rule`:**

| Attribute | 2016 | 2020 |
|---|---|---|
| `id` | `xsd:ID` | `xsd:NCName` |
| `name` | `text` | `text` |
| `context` | `text` | `text` |
| `role` | `xsd:NMTOKEN` | `string` |
| `flag` | `xsd:NMTOKEN` | `string` |
| `document` | *(not present)* | **`xsd:anyURI`** (new, optional) |

**`attlist.assert-and-report`:**

| Attribute | 2016 | 2020 |
|---|---|---|
| `id` | `xsd:ID` | `xsd:NCName` |
| `location` | `text` | `text` |
| `test` | `text` | `text` |
| `role` | `xsd:NMTOKEN` | `string` |
| `flag` | `xsd:NMTOKEN` | `string` |

The shift `xsd:ID` → `xsd:NCName` for `id` attributes throughout the SVRL grammar means that repeated `id` values across an SVRL document are no longer flagged by the grammar (because `xsd:NCName` does not impose uniqueness). This is a relaxation likely motivated by the fact that SVRL is a *report*, where the same rule firing in multiple contexts could otherwise force the generator to mint fresh IDs.

The shift `xsd:NMTOKEN` → `string` for `role` / `flag` lifts the lexical restriction that these values be NMTOKENs, allowing whitespace and richer text.

The new optional `document` attribute on `fired-rule` is significant — it allows an SVRL document to record which subordinate document a rule fired against, complementing the `documents` attribute on `pattern`.

## 7 Migration notes

Practical impact of moving an implementation from ISO/IEC 19757-3:2016 to ISO/IEC 19757-3:2020:

- **No element/attribute additions in the core grammar** — existing Schematron parsers and validators that target the 2016 grammar will accept all 2020 schemas without code change.
- **Schema-level `properties` is now optional** — if your validator required `properties` to be present on `schema` because the 2016 RNC said so, relax that check (2020 marks it `properties?`). This is a behaviour-aligning fix; most implementations were already lenient here.
- **`diagnostic` may now carry an explicit `role` attribute** — extend SVRL/diagnostic post-processing to surface `diagnostic/@role` ("warning", "caution", "note", …).
- **New reserved `queryBinding` values**: implementations should accept `xslt3`, `xpath3`, `xpath31`, `xquery3`, `xquery31` as defined identifiers (in addition to the 2016 set `stx`, `xslt`, `exslt`, `xslt2`, `xpath`, `xpath2`, `xquery`).
- **First-class XSLT 3.0 / XPath 3.0 bindings** (Annexes J and K) — implementers wishing to claim conformance to one of these bindings now have a normative reference. The semantics mirror the XSLT2/XPath2 bindings, so implementations using Saxon-HE 10.x+ or similar XSLT3 engines can claim conformance straightforwardly.
- **XPath 2.0 binding (Annex I) is now normative** — implementations claiming an `xpath2` binding are now subject to normative conformance requirements rather than informative guidance.
- **SVRL changes** that may affect generators or consumers:
  - Generators may emit a new `document` attribute on `fired-rule` to report which subordinate document the rule fired against (matches the pattern-level `documents` attribute).
  - Generators are no longer constrained to mint globally unique `id`s in SVRL (`xsd:ID` → `xsd:NCName`).
  - Generators may now emit `role` / `flag` values that contain whitespace or other non-NMTOKEN characters (`xsd:NMTOKEN` → `string`). Consumers should not assume NMTOKEN lexical form.
  - The `schematron-output` content model now allows an active pattern with no fired rules — generators no longer need to skip empty active patterns.
- **Annex B (additional constraints) is now also part of simple conformance** — simple-conformance implementations must reject schemas that fail Annex B's Schematron rules (was 2016 full-conformance only).
- **Conformance wording for variable scoping is sharper**: implementations performing the global-only "one definition for any variable name" check in 2016 should now distinguish global-context and local-context scopes per the new wording.
- **`ns` element semantics tightened (SHOULD → SHALL)** — implementations should now reject (or at least warn loudly about) Schematron schemas that rely on `xmlns:` declarations rather than `sch:ns` to resolve prefixes used in queries.
- **`6.2 Minimal syntax` is informative** — implementations that previously felt obliged to transform schemas to "minimal syntax" before evaluation no longer need to; this is presentation guidance only.
- **Drop of the term `compound document`** — purely terminological; no implementation impact (the concept is retained in Annex E).
- **Removed NOTE markers about ISO/IEC 19757-3:2006 differences** — purely editorial; implementations should not rely on those markers being present.

> **Caveat** — the 2020 §7.2 (Full conformance) still uses the term "good schema" even though the term is no longer defined in §3. Implementations claiming full conformance should continue to apply the 2016 definition of *good schema* (i.e. a correct schema whose queries terminate and do not add constraints to those of the natural-language assertions). This is an editorial inconsistency in the 2020 edition rather than a substantive change.

> **Caveat** — the 2020 §3.22 definition of *valid* uses the text `<schema>` in markup, which appears to be a layout artifact in the published edition. The intended meaning is "document passing all assertion tests in fired rules of active patterns" (consistent with 2016 §3.25).
