# Delta: ISO/IEC 19757-3:2006 to ISO/IEC 19757-3:2016

**Comparison of:**

- **v1 (source A):** ISO/IEC 19757-3:2006 — First edition, 2006-06-01 — `c040833_ISO_IEC_19757-3_2006.md`
- **v2 (target B):** ISO/IEC 19757-3:2016 — Second edition, 2016-01-15 — `c055982_ISO_IEC_19757-3_2016.md`

**Delta authored:** 2026-06-06

---

## High-level summary

The 2016 edition is a **technical revision** of the 2006 first edition. The Foreword of v2 states explicitly that "This second edition cancels and replaces the first edition (ISO/IEC 19757-3:2006), which has been technically revised." The Introduction further declares that the edition is **backwards compatible** with the first edition and **augments** it with:

- A facility for patterns to validate **different (subordinate) documents** via a new `documents` attribute on `pattern`.
- An **enhanced extension mechanism** for the `extends` element (an `href` attribute to pull in external XML fragments) complementing the existing `include` mechanism.
- A **properties feature** — assertions and reports can have linked metadata via new `properties` / `property` elements and a `properties` attribute on `assert`/`report`.
- A **richer SVRL text model** (the SVRL `text` element now allows attributes and rich inline markup such as `dir`, `emph`, `span` and foreign content; a new `property-reference` element).
- **Additional query language bindings**: a normative XSLT 2 binding and informative bindings for XPath 2, EXSLT and STX.

Dominant themes of change:

1. **New query bindings** — XSLT 2 (normative Annex H), XPath 2, EXSLT, STX (informative Annexes I, J, K).
2. **Schematron properties** — a new ancillary concept for attaching named metadata to assertions and reports (Annex L is a new informative annex of examples).
3. **Compound documents** — a new term "compound document" plus the `documents` attribute on `pattern` enable validation across linked sub-documents.
4. **Inclusion / extension model** — `extends` gains an `href` alternative to its `rule` reference; `include` is re-specified in terms of IRI references that may target an element within a document.
5. **SVRL enrichment** — `text` becomes a structured rich-text element, `property-reference` is added, `active-pattern` and `fired-rule` gain a `documents` / `name` attribute, the human-text wrapper is overhauled.
6. **Editorial / structural tightening** — section 4 (Notation) is collapsed (predicate-logic subclause removed); section 6.3 is renamed and the formal predicate-logic semantics block is dropped from clause 6; terminology is renumbered.
7. **Normative status changes** for several annexes (Annex F drops from normative to informative; Annex H is the new normative annex).

The change is conservative for existing 2006-conformant schemas: nothing in the 2006 element set has been removed in 2016, and existing schemas continue to validate against the updated Annex A grammar.

## RNC schema delta (Annex A)

The Annex A schemas of v1 and v2 are largely the same, but v2 introduces five concrete grammar changes plus formatting/whitespace differences.

### 1. `schema` — `properties` child added at end of content model

v1:

```rnc
schema = element schema {
    attribute id { xsd:ID }?,
    rich,
    attribute schemaVersion { non-empty-string }?,
    attribute defaultPhase { xsd:IDREF }?,
    attribute queryBinding { non-empty-string }?,
    (foreign
     & inclusion*
     & (title?, ns*, p*, let*, phase*, pattern+, p*, diagnostics?))
}
```

v2:

```rnc
schema =
  element schema {
    attribute id { xsd:ID }?,
    rich,
    attribute schemaVersion { non-empty-string }?,
    attribute defaultPhase { xsd:IDREF }?,
    attribute queryBinding { non-empty-string }?,
    (foreign
     & inclusion*
     & (title?, ns*, p*, let*, phase*, pattern+, p*, diagnostics?, properties))
  }
```

Note that `properties` appears at the tail of the sequence and — unlike `diagnostics?` — is shown **without a `?`** in v2's printed grammar. Strictly read, this would make a `properties` element mandatory on every schema, which would break backward compatibility; this is almost certainly an editorial slip in the published RNC (the prose text of 5.5.9 treats `properties` as optional, and the rest of the standard is declared backwards compatible). Implementers should treat `properties` as optional.

### 2. `assert` / `report` — new `properties` IDREFS attribute

v1 `assert`:

```rnc
attribute test { exprValue },
attribute flag { flagValue }?,
attribute id { xsd:ID }?,
attribute diagnostics { xsd:IDREFS }?,
rich, linkable, ...
```

v2 `assert` (and identically `report`) adds:

```rnc
attribute properties { xsd:IDREFS }?,
```

### 3. `extends` — `rule` OR `href` choice

v1:

```rnc
extends = element extends {
    attribute rule { xsd:IDREF },
    foreign-empty
}
```

v2:

```rnc
extends =
  element extends {
    (attribute rule { xsd:IDREF }
     | attribute href { uriValue }),
    foreign-empty
  }
```

### 4. `let` — `value` attribute OR foreign-element content

v1:

```rnc
let = element let {
    attribute name { nameValue },
    attribute value { string }
}
```

v2:

```rnc
let =
  element let {
    attribute name { nameValue },
    (attribute value { string }
     | foreign-element+)
  }
```

### 5. `pattern` — new `documents` attribute

v1's `pattern` body is unchanged; v2 adds:

```rnc
attribute documents { pathValue }?,
```

immediately before `rich`.

### 6. `rule` body — `p` element now permitted

v1:

```rnc
(assert | report | extends)+
```

v2:

```rnc
(assert | report | extends | p)+
```

(applies to both abstract and non-abstract branches).

### 7. `inclusion` — accepts foreign attributes

v1:

```rnc
inclusion = element include {
    attribute href { uriValue }
}
```

v2:

```rnc
inclusion =
  element include {
    attribute href { uriValue },
    foreign-empty
  }
```

### 8. New `properties` and `property` productions

v2 adds:

```rnc
properties = element properties { property* }

property =
  element property {
    attribute id { xsd:ID },
    attribute role { roleValue }?,
    attribute scheme { text }?,
    (foreign & (text | name | value-of | emph | dir | span)*)
  }
```

### 9. Common declarations and datatypes

The `rich`, `linkable`, `foreign`, `foreign-empty`, `foreign-attributes`, `foreign-element` definitions and the datatype primitives (`uriValue`, `pathValue`, `exprValue`, `fpiValue`, `langValue`, `roleValue`, `flagValue`, `nameValue`, `classValue`, `non-empty-string`) are **unchanged**.

### 10. Header / copyright comment

v1 has a brief ISO 2005 grant note; v2 adopts the longer ISO 2015 MIT-style permission notice and disclaimer (about 18 lines of comment).

## Annex delta

### Annex A (normative, RNC schema)

Updated as detailed in the [RNC schema delta](#rnc-schema-delta-annex-a) section above.

### Annex B (normative, additional constraints as a Schematron schema)

Substantively unchanged. The four rules (on `sch:active`, `sch:pattern[@is-a]`, `sch:extends`, `sch:let`) are character-for-character identical apart from whitespace/indentation reflow. The copyright header is updated from ISO 2005 to the ISO 2015 MIT-style notice.

### Annex C (normative, default query language binding)

**Added**

- A bullet covering the `documents` attribute on `pattern`: it "shall be interpreted according to the Production 1 of XSLT1, as returning one or more IRI strings that are evaluated using the `document()` function".
- A new permission: the **`copy-of` element from the XSLT1 namespace** may be used (without child nodes) inside the `property` element.
- The rule context bullet is extended: "An implementation may allow the rule context to be text nodes at user option however implementations may reject or fail to implement schemas which specify text nodes."

**Changed**

- All references to "XSLT" are qualified as "XSLT1" throughout.
- The phrase "queryBinding attribute" is replaced with "language attribute" in the heading sentence (apparent editorial slip).

**Removed**

- Nothing of substance.

### Annex D (informative, SVRL)

**Added (RNC)**

- New `property-reference` element production:

  ```rnc
  property-reference =
    element property_reference {
      attribute property { xsd:NMTOKEN },
      attribute role { text }?,
      attribute scheme { text }?,
      human-text
    }
  ```

- New `documents` attribute on `active-pattern`.
- New `name` attribute on `fired-rule` and on `active-pattern`.
- The `human-text` wrapper (`element text`) gains attributes `xml:space`, `xml:lang`, `see`, `icon`, `fpi` and now contains a `rich-text` body (mix of foreign / `dir` / `span` / `emph` / text). Previously it was just `element text { text }`.
- New `rich-text` content model and reusable `dir` / `emph` / `span` element productions in the SVRL namespace, each accepting a `class` attribute.
- `foreign` declarations are added (mirroring Annex A).
- `failed-assert` and `successful-report` now include `property-reference*` in their content model alongside `diagnostic-reference*` and `human-text`.

**Changed**

- Copyright header updated to ISO 2014/2015 MIT-style permission notice.
- The corresponding "Schema for SVRL" Schematron schema in D.3 is preserved in structure but reformatted; it does **not** yet include rules for the new `property-reference` element (an inconsistency between D.2 and D.3 in the published v2 text).
- A minor known XPath bug in v2 D.3: `<sch:assert test="not(preceding::*[@id=current()/@id][1]">` — missing closing `)`.

**Removed**

- Nothing structural was removed from the v1 SVRL schema.

### Annex E (informative, design requirements)

**Added**

- An additional motivation bullet "Support progressive validation of compound documents" (v1 just said "Support progressive validation").
- A new closing list: "Motivating requirements for the changes introduced in this part of ISO/IEC 19757 include the following" with two bullets: "Support more query language bindings, in particular XSLT2" and "Support the declaration of simple static and dynamic properties".

### Annex F (use of Schematron as a vocabulary)

**Changed**

- Status downgraded from **normative** (v1) to **informative** (v2). Content is otherwise identical.

### Annex G (informative, multi-lingual schemas)

Unchanged in content; minor reformatting only.

### Annex H (normative) — Query language binding for XSLT 2 — **NEW**

A full normative annex defining the `xslt2` binding. Key provisions:

- Query language is XPath 2.0 as extended by XSLT 2.0 with backwards-compatibility mode false; data model is XDM.
- All XPath 2 functions and operators available; implementations may allow user-written functions.
- Rule context interpreted by XSLT2 Production 1; may be root nodes / elements / attributes / comments / processing instructions including sequences. Text nodes optional.
- Assertion test uses `fn:boolean()`; name query uses `fn:node-name()`; value-of uses `fn:string()`.
- `documents` attribute uses XSLT2 Production 1 returning a sequence of strings passed to `document()`.
- The XSLT2 `key`, `function`, and `copy-of` elements may be used (the latter inside `property`).
- All namespaces/prefixes defined by XSLT2 modules are reserved.
- `fn:error()` and other dynamic errors **should not** be used to substitute for assertion text.
- The annex states "all implementations that support `xslt2` should also support `xpath2`".

### Annex I (informative) — Query language binding for XPath 2 — **NEW**

Defines the `xpath2` binding (XPath 2 used standalone, no XSD/PSVI, no user-written functions, backwards-compatibility mode false). Rule context limited to elements, attributes, comments and processing instructions. `let` element should not be used. Same `fn:error()` caveat.

### Annex J (informative) — Query language binding for EXSLT — **NEW**

Defines `exslt` as the default `xslt` binding augmented by EXSLT functions. All EXSLT namespaces/prefixes are reserved. Includes the `documents` attribute interpretation and `key` element permission. NOTE: not all EXSLT modules are required of every implementation.

### Annex K (informative) — Query language binding for STX — **NEW**

Defines `stx` binding using the streaming dialect of XPath specified in STX. All elements/attributes are treated as untyped. `message` element shall not be used for assertion text.

### Annex L (informative) — Example usage of Schematron properties — **NEW**

A new informative annex giving eight worked examples (L.1–L.9) demonstrating use of `properties`/`property`:

- L.2 Annotation with controlled vocabulary information (`<sch:property id="australianDollar" scheme="ISO 4217" role="currency">AUD</sch:property>`).
- L.3 Static information.
- L.4 Static structured information (nested non-Schematron element content).
- L.5 Type annotation via XSD Simple Type (`xsd:simpleType` inside `property`).
- L.6 Dynamic information using `sch:value-of`.
- L.7 Dynamic structured information using `xslt:copy-of`.
- L.8 ISO/IEC 19757-7 character repertoire reference.
- L.9 Example SVRL document showing how `svrl:property-reference` carries the property values across.

## Migration notes

Practical impacts for a Schematron implementer upgrading from a 2006-conformant implementation to a 2016-conformant one:

- **All v1 schemas remain valid in v2.** The 2016 edition is explicitly backwards compatible; existing core/ancillary elements and attributes are preserved. No deprecations.

- **New optional `documents` attribute on `pattern`.** Implementers must decide how to support it; in the default XSLT1 binding the value is an XPath expression evaluated in the *original instance root* context returning one or more IRI strings that are then dereferenced via XSLT's `document()`. The pattern is applied to each subordinate document. SVRL `active-pattern` gains a matching `documents` attribute to report which subordinate was used.

- **`extends/@href` enables external fragment inclusion.** Implementations that previously only supported `extends/@rule` must add support for resolving an `href` (IRI), retrieving an XML document or a single addressed element within it, and splicing the children of that element in place of the `extends` element. Note v2 5.4.3 states "the relative position of elements in the post-inclusion document may be to that extent invalid against the schema for Schematron Annex A; however, other schema constraints such as containment shall still apply" — a relaxation of the structural validity rule for included fragments.

- **`include/@href` may now target a sub-element.** Update IRI resolution to honour fragment identifiers that pick a particular element from within a document.

- **`let` without `value` attribute.** When a `let` element has no `value` attribute, treat its element-content (a sequence of foreign elements) as the value. This is useful for embedding non-string typed payloads in XSLT 2 and other rich bindings.

- **Properties feature** — three new things to support:
  1. Recognise `sch:properties` (container) as a child of `sch:schema` and `sch:property` (id-bearing item with `role`/`scheme`/mixed content).
  2. Recognise the `properties` IDREFS attribute on `sch:assert` and `sch:report`.
  3. When producing SVRL, emit `svrl:property-reference` children of `failed-assert`/`successful-report` carrying the property `id`, optional `role`, optional `scheme` and the evaluated/literal content. In the XSLT1 and XSLT2 default bindings, `xslt:copy-of` (without children) is permitted inside `sch:property` for dynamic content; in XSLT2 the `xslt2:copy-of` is supported.

- **Allow `p` as child of `rule`.** The RNC body for `rule` now permits `p` alongside `assert`/`report`/`extends` in both abstract and non-abstract rules. Parsers / SVRL generators must not reject this.

- **SVRL output expansion** — implementers generating SVRL need:
  - To emit the new `documents` attribute on `active-pattern` when the underlying pattern declared one.
  - To emit the new `name` attribute on `active-pattern` and `fired-rule`.
  - To support the richer `text` model: `text` now allows `xml:space`, `xml:lang`, `see`, `icon`, `fpi` attributes and rich inline content (`dir`, `emph`, `span`, foreign elements) instead of plain `text`.
  - To support `property-reference` emission.
  - The SVRL `dir`, `emph`, `span` elements now allow a `class` attribute (and `dir` an additional `dir` attribute).

- **Query bindings**:
  - The reserved name `xslt1.1` is **removed** from the list of reserved query language names. Other reserved names (`stx`, `xslt`, `exslt`, `xslt2`, `xpath`, `xpath2`, `xquery`) are unchanged.
  - A normative XSLT 2 binding (`xslt2`) is now defined — implementations targeting XSLT 2 should follow Annex H. Notably, `fn:error()` should not be used for assertion text.
  - Informative bindings for `xpath2`, `exslt`, `stx` are now described. They are non-normative but provide a baseline for interoperable implementations.

- **Compound documents** are formally introduced as a term (3.7); validation can now span the original instance plus any number of subordinate documents reached via `pattern/@documents`.

- **No predicate-logic semantics**. The formal `∀ (context, pattern, rule, assertion):…` semantics block from v1 6.3 is gone in v2. The normative anchor for validity is now (a) the term 3.25 "valid with respect to a schema", (b) the minimal-syntax transformation in 6.2, and (c) the order rules in 6.5. Implementations relying on the formal v1 statement should re-anchor on the prose definitions.

- **Annex F now informative.** Use of the five top-level Schematron element types (`schema`, `pattern`, `rule`, `assert`, `report`) as vocabulary elements in other schemas is no longer a normative requirement; it remains a recommendation.

- **References update.** Annex A and the SVRL schema headers now carry an ISO 2015 MIT-style permission notice instead of the 2005 SGML-systems grant.

### Editorial irregularities to be aware of in the v2 published text

While reviewing the standard line by line for this delta, the following published-text irregularities were observed (none change normative intent but implementers should be aware):

- v2 5.3 whitespace listing types `U+009`, `U+00D`, `U+00A` instead of the correct `U+0009`, `U+000D`, `U+000A`.
- v2 5.4.2 `assert` element says "If the test evaluates positive, the **report** succeeds" (carryover from `report` description).
- v2 Annex A `schema` content model lists `properties` without `?`, which would mandate the element. The accompanying prose (5.5.9) treats `properties` as optional and the foreword promises backwards compatibility.
- v2 Annex C heading sentence says "language attribute" where it should say "`queryBinding` attribute".
- v2 Annex D RNC element name `property_reference` (underscore) vs. the rest of the spec using `property-reference` (hyphen). The XML examples in Annex L use `property-reference` so the underscore is presumed an editorial slip.
- v2 Annex D RNC `default namespace svrl = http://purl.oclc.org/dsdl/svrl` lacks the surrounding quotes that valid RNC requires.
- v2 Annex L's running XML example has an unclosed `<sch:rule>`/`<sch:pattern>` nesting (`</sch:pattern>` directly closes a `rule`).
- v2 Annex L.9's example SVRL uses mismatched closing tags (`</sch:property>` closing a `<svrl:property-reference>` and `</property>` closing another `<svrl:property-reference>`).

These are presented for context; none are normative changes from v1 to v2.
